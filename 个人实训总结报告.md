# 个人实训总结报告

## 一、项目概述

本次实训我独立完成了一个"习惯打卡管理系统（Habit Tracker）"，这是一个基于 Next.js 14 全栈框架的 Web 应用。项目包含3个前端页面（今日打卡、习惯管理、数据统计）和3个后端 API 接口（获取列表、创建习惯、打卡操作），最终部署在 Vercel 平台上，可通过线上链接直接访问。

## 二、遇到的最大困难及解决过程

在开发过程中，我遇到了一个主要的困难：**数据存储方案的选择**。最初我考虑使用 MongoDB 或 PostgreSQL 等数据库，但部署到 Vercel 上需要额外的数据库服务配置，对于一个小型打卡系统来说过于复杂。经过思考和 AI 工具的辅助分析，我最终选择了本地 JSON 文件存储方案。这种方式虽然不支持并发写入，但对于个人使用的打卡系统完全够用，而且零配置即可在 Vercel 上运行。

另一个困难是 **TypeScript 类型检查**。在开发统计页面时，我使用了 `Set` 数据结构来快速查找打卡记录，但在构建时遇到了 `Set` 展开运算符的兼容性报错。通过查阅 Next.js 的 TypeScript 配置，我了解到需要使用 `Array.from()` 替代展开运算符来解决 `downlevelIteration` 的问题。

## 三、对 AI 辅助编程的心得体会

这次实训让我深刻体会到了 AI 辅助编程的强大之处。通过 Trae 工具，我可以用自然语言描述需求，AI 能够快速生成完整的代码框架和组件。但我也认识到，AI 并不是"万能的"——它生成的代码有时会存在 ESLint 未使用变量的报错、TypeScript 类型不匹配等问题，需要人工审查和调试。

我认为 AI 辅助编程的正确方式是：**AI 负责快速生成基础代码，人类负责审查、调试和完善**。在这个过程中，Git 的提交记录变得尤为重要——它不仅是"考勤表"，更是开发过程的完整记录。每次提交都反映了我在 AI 生成代码基础上的思考和改进。

## 四、个人在项目中的具体贡献

本项目由我独立完成，具体工作包括：
1. **需求分析与技术选型**：确定使用 Next.js 全栈方案，设计数据模型
2. **后端开发**：实现了3个 API 接口，包含完整的参数校验和错误处理
3. **前端开发**：完成了3个页面的 UI 设计和交互逻辑，包括打卡 Toggle、热力图、进度条等
4. **工程化实践**：配置了 TypeScript 类型系统、ESLint 代码规范、规范的 Git 提交记录
5. **文档编写**：撰写了 README.md、API_DOCS.md、prompt_log.md 等完整文档
6. **部署上线**：将项目部署到 Vercel 平台，获得线上访问链接

## 五、总结

通过本次实训，我不仅掌握了一个完整的前后端项目开发流程，更重要的是学会了如何高效地利用 AI 工具提升开发效率，同时保持对代码质量的把控。这种"人机协作"的开发模式，我认为代表了未来软件开发的一个重要方向。
